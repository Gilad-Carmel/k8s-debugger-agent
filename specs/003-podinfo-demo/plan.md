## Summary

Deploy [podinfo](https://github.com/stefanprodan/podinfo) as a controlled demo
workload and provide four one-command scenario scripts that trigger failure modes,
fire a real Alertmanager-format webhook, and drive the full triage+approval+remediation
workflow end-to-end against a live kind cluster.

No agent or MCP server code is modified. All new files live in `deploy/demo/` and
`scripts/demo/`.

## Implementation Phases

### Phase 1: Manifests (no-code)

1. `deploy/demo/00-namespace.yaml` — demo namespace with `purpose: demo` label
2. `deploy/demo/01-podinfo-v1.yaml` — v1 Deployment, 32Mi memory limit, liveness + readiness probes, `app: podinfo` label
3. `deploy/demo/02-podinfo-v2-bad.yaml` — v2 patch overlay with `PODINFO_RUNTIME_ERROR=true`
4. `deploy/demo/03-service.yaml` — ClusterIP on port 9898
5. `deploy/demo/04-pdb.yaml` — PDB with `minAvailable: 1`

### Phase 2: Core scripts

6. `scripts/demo/deploy.sh` — `kubectl apply -f deploy/demo/`, wait for Ready (`kubectl rollout status`)
7. `scripts/demo/teardown.sh` — `kubectl delete namespace demo --ignore-not-found`
8. `scripts/demo/fire-webhook.sh`:
   - Accepts `--scenario`, `--namespace`, `--pod`, `--agent-url`, `--hmac-secret`
   - Constructs Alertmanager v4 JSON body
   - Computes HMAC-SHA256 signature: `echo -n "<body>" | openssl dgst -sha256 -hmac "$SECRET"`
   - Sets `X-Alertmanager-Hmac: sha256=<hex>` header
   - POSTs to `$AGENT_WEBHOOK_URL` and prints response + `correlation_id`

### Phase 3: Scenario trigger scripts

Each script:
1. Starts `kubectl port-forward svc/podinfo 9898:9898 -n demo &` → saves PID
2. Triggers the failure
3. Polls until failure is observable (state check loop, max 30s)
4. Calls `fire-webhook.sh` with appropriate `--scenario`
5. Kills port-forward PID
6. Prints the `correlation_id`

9. `scripts/demo/trigger-crash.sh`
10. `scripts/demo/trigger-bad-deploy.sh`
11. `scripts/demo/trigger-oom.sh`
12. `scripts/demo/trigger-scale.sh`

### Phase 4: Makefile targets

Append to `Makefile`:

```makefile
.PHONY: demo-deploy demo-teardown demo-reset demo-crash demo-bad-deploy demo-oom demo-scale

demo-deploy:
	@bash scripts/demo/deploy.sh

demo-teardown:
	@bash scripts/demo/teardown.sh

demo-reset: demo-teardown demo-deploy

demo-crash: demo-deploy
	@bash scripts/demo/trigger-crash.sh

demo-bad-deploy: demo-deploy
	@bash scripts/demo/trigger-bad-deploy.sh

demo-oom: demo-deploy
	@bash scripts/demo/trigger-oom.sh

demo-scale: demo-deploy
	@bash scripts/demo/trigger-scale.sh
```

### Phase 5: Smoke test

13. `tests/integration/test_demo_scenarios.sh`:
    - `make demo-deploy` → assert pod becomes Ready
    - `fire-webhook.sh --scenario crash-loop --dry-run` → assert JSON is valid
    - `make demo-teardown` → assert namespace gone

## Key Design Decisions

### Why synthetic webhooks instead of real Alertmanager?

Installing Prometheus + Alertmanager into a kind cluster adds ~500MB and 5+
minutes of setup time. The demo's value is the triage workflow, not the alerting
stack. A shell script that constructs a valid webhook payload exercises the
real ingestion path (HMAC validation, dedup, correlation ID) without the overhead.

### Why 32Mi memory limit on podinfo?

`/stress?mem=50` allocates 50Mi. A 32Mi limit guarantees OOMKill within a few
seconds of the stress call, making the OOM scenario fast and deterministic.

### Why a dedicated `demo` namespace?

Keeps demo state isolated. `make demo-teardown` (`kubectl delete namespace demo`)
is a complete, safe cleanup with no risk of touching system workloads.
